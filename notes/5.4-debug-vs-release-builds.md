# Module 5.4 – Debug vs Release Builds

### Learning Goals

- Understand the difference between **debug builds** and **release builds**
- Learn why softeare is compiled with different configurations
- Understand the role of optimization flags
- Learn how debug symbols work
- Understand how assertions and sanitizers behave in different builds
- Learn the typical build configurations used in professional C projects

---

## The Core Idea

A build configuration determines how the compiler translates source code into an executable.

The same program can be compiled in different ways depending on the goals:

| Build Type | Goal |
|------------|------|
| Debug | Detect bugs and simplify debugging |
| Release build | Maximize performance and efficiency |

Because of this, developers usually maintain **multiple build configurations**.

---

## Debug Build

A **debug build** prioritizes:

- correctness
- debuggability
- bug detection

Typical flags:

```
-O0
-g
-fsanitize=address,undefined
```

Meaning:

| Flag | Purpose |
|------|---------|
| `-O0` | Disable optimizations |
| `-g` | Include debug symbols |
| `-fsanitize` | Detect runtime errors |

---

## Why Disable Optimizations?

Compiler optimizations can:

- reorder instructions
- eliminate variables
- inline functions
- remove code the compiler thinks is unnecessary

This makes debugging difficult.

Example:

```c
int x = 5;
int y = x + 1;
```

With optimizations:

```
y = 6
```

The compiler may eliminate `x` entirely.

A debugger may show:

```
x = <optimized out>
```

This is confusing when debugging.

---

## Debug Symbols

The `-g` flag tells the compiler to include **debugging information** in the executable.

This allows debuggers (like `gdb` or `lldb`) to map:

```
machine instructions → source code lines
```

Without `-g`, a debugger only sees machine addresses.

---

## Sanitizers

Sanitizers are runtime tools that detect bugs.

Example:

```
-fsanitize=address
```

Detects:

- buffer overflows
- use-after-free
- double free
- invalid memory access

Example: 

```
-fsanitize=undefined
```

Detects:

- integer overflow
- invalid shifts
- null dereference
- undefined operations

These tools are **extremely valuable during development**.

---

## Release Build

A **release build** prioritizes:

- performance
- small binary size
- optimized execution

Typical flags:

```
-O2
-DNDEBUG
```

Meaning:

| Flag | Purpose |
|------|---------|
| `-O2` | Enable compiler optimizations |
| `-DNDEBUG` | Disable assertions |

Release builds remove debugging overhead.

---

## Assertions in Release Builds

Recall:

```c
assert(x > 0);
```

If the program is compiled with:

```
-DNDEBUG
```

The assertion becomes:

```
((void)0)
```

Meaning the assertion **completely disappears**.

This avoids runtime cost in production.

---

## Typical Build Configurations

Professional C projects usually define two configurations.

#### Debug

```
-std=c17
-Wall
-Wextra
-Werror
-O0
-g
-fsanitize=address,undefined
```

Purpose:

> detect bugs early

#### Release

```
-std=c17
-Wall
-Wextra
-O2
-DNDEBUG
```

Purpose:

> maximum performance

---

## Why Bugs appear only in Release Builds

This is a famous C problem.

Example: 

```c
int x;
if (x == 0) {
    do_something();
}
```

`x` is uninitialized.

In debug builds, memory may happen to be zero.

In release builds, optimizations may produce completely different behavior.

This is why **UB often appears only in optimized builds**.

---

## Professional Practice

Most projects automate build modes:

Example Makefile pattern:

```
make debug
make release
```

Debug:

```
-O0 -g -fsanitize
```

Release:

```
-O2 -DNDEBUG
```

This ensures the correct configuration for each stage.

---

## Connection to the `csvstat` project

The `csvstat` project uses something like:

```
-std=c17
-Wall
-Wextra
-Werror
-O0
-g
-fsanitize=address,undefined
```

That is a **debug build configuration**, which is exactly what we want during development.

---

## Takeaway

> - **Debug builds** prioritize **bug detection and debugging**.  
> - **Release builds** prioritize **performance and optimization**.  
> - `-O0` simplifies debugging by *disabling optimizations*.  
> - `-g` enables *source-level debugging*.  
> - Sanitizers detect memory and UB errors at runtime.  
> - `-DNDEBUG` removes assertions in release builds.  
> - Professional projects maintain **separate debug and release configurations**.
