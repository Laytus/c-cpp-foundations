# Module 5.2 – UB Patterns and How to Avoid Them

### Learning Goals

- Understand what **Undefined Behavior (UB)** means in C
- Learn why UB is extremely dangerous
- Recognize common UB patterns
- Understand how compilers exploit UB for optimizations
- Learn techniques to prevent UB
- Use tooling to detect UB

---

## The Core Idea

**Undefined Behaviro (UB)** occurs when a C program executes an operation for which the C standard **does not define the result**.

When UB happens:

```
Anything can happen.
```

Possible outcomes include:

- program crashes
- incorrect results
- security vulnerabilities
- program appears to work
- different behavior on different machines
- different behavior under optimization

The compiler is allowed to **assume UB never happens**, which allows aggressive optimizations.

This is why UB bugs can be extremely difficult to diagnose.

---

## Why UB exists

C was designed for:

- operation systems
- embedded systems
- high performance
- hardware-level control

To achieve this, the language avoids runtime checks.

Instead of defining behavior for every possible mistake, the standard simply says:

> behavior is undefined

This keeps the language **fast and simple**, but places responsability on the programmer.

---

## Common UB Patterns

#### Using Unitialized Variables

Example:

```c
int x;
printf("%d\n", x);
```

`x` was never initialized.

Result:

```
Undefined Behavior
```

Possible outputs:

```
0
42
-928374
```

Or program crash.

#### Dereferencing Invalid Pointers

Example:

```c
int *p = NULL;
*p = 5;
```

This dereferences a null pointer.

Result:

```
Undefined Behavior
```

Usually leads to **segmentation fault**.

#### Out-of-Bounds Array Access

Example:

```c
int a[3] = {1,2,3};
printf("%d\n", a[5]);
```

Accessing outside array bounds.

This may read:

- unrelated memory
- stack variables
- invalid memory

#### Use-After-Free

Example:

```c
int *p = malloc(sizeof(int));
free(p);
*p = 10;
```

After freeing memory, the pointer becomes invalid.

Accessing it produces UB.

#### Double Free

Example:

```c
free(p);
free(p);
```

Freeing the same memory twice corrupts the **heap**.

#### Signed Integer Overflow

Example:

```c
int x = INT_MAX;
x = x + 1;
```

Signed overflow is **undefined behavior** in C.

Many developers assume wrap-around happens – but that is **not guaranteed**.

#### Modifying a variable multiple times in one expression

Example:

```c
i = i++ + 1;
```

This modifies `i` multiple times without sequencing.

---

## Why UB is dangerous

The compielr may optimize based on the assumption:

```
Undefined behavior never happens.
```

Example:

```c
if (p != NULL)
    *p = 10;
```

If the compiler proves `p` is never null earlier in the code, it may remove the check entirely.

This can cause **unexpected behavior**.

---

## Professional Techniques to avoid UB

#### Always Initialize Variables

Bad:

```c
int x;
```

Good:

```c
int x = 0;
```

#### Set Pointers to NULL After Free

Example:

```c
free(p);
p = NULL;
```

This prevents accidental reuse.

#### Use Array Bounds carefully

Example:

```c
for (size_t i = 0; i < n; i++)
```

Never rely on guessed limits.

#### Avoid Complex Expressions

Bad:

```c
a[i++] = i;
```

Good: 

```c
a[i] = i;
i++;
```

Clear sequencing avoids UB.

#### Use Sanitizers

Example:

```bash
-fsanitizers=address,undefined
```

They detect:

- out-of-bounds access
- use-after-free
- integer overflow
- invalid pointer operations

---

## Real Industry Practice

Serious C projects often run:

```
sanitizers
static analysis
fuzz testing
```

to detect UB.

Even large projects like:

- Linux kernel
- SQLite
- Firefox

regularly fix UB bugs discovered by tooling.

---

## Practical Rule

When writing C code, constantly ask:

> Could this produce undefined behavior?

If the answer might be yes, redesign the code.

---

## Takeaway

> - **Undefined Behavior** means the C standard defines **no result**.  
> - UB can produce *crashes, corruption, or silent bugs*.  
> - Common UB sources include *uninitialized variables, invalid pointes, out-of-bounds access, and signed overflow*.  
> - Compilers assume UB never occurs and optimize accordingly.  
> - Defensive programming and tooling help prevent UB.  
> - Always design code so UB **cannot occur**.
