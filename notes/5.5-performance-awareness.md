## The Core Idea

In modern computers, the **main performance bottleneck is usually memory access**, not computation.

CPUs can perform billions of arithmetic operations per second.

But *accessing memory* is **much slower**.

Approximate access times:

| Storage | Latency |
|---------|---------|
| CPU register | ~1 cycle |
| L1 cache | ~4 cycles |
| L2 cache | ~12 cycles |
| L3 cache | ~40 cycles |
| RAM | ~100–300 cycles |

Because of this:

> Programs that access memory efficiently are often much faster than those that do not.

Performance awareness means writing code that **works well with the hardware**, especially the CPU cache.

---

## Cache Locality

CPU caches store recently used memory to avoid expensive RAM accesses.

Two types of locality matter:

| Type | Meaning |
|------|---------|
| Temporal locality | Recently used data is likely to be used agan |
| Spatial locality | Nearby memory locations are likely to be used |

Programs that access memory **sequentially** benefit greatly from caching.

---

## Example: Sequential vs Random Access

#### Cache-friendly access

```c
for (size_t i = 0; i < n; i++) {
    sum += array[i];
}
```

Memory is accessed sequentially.

This works well with CPU caches.

#### Cache-unfriendly access

```c
for (size_t i = 0; i < n; i++) {
    sum += array[random_index[i]];
}
```

Memory is accessed unpredictably.

This causes frequent **cache misses**, which slows down execution.

---

## Example: Row-Major Arrays

C stores multi-dimensional arrays in **row-major order**.

Example: 

```c
int matrix[1000][1000];
```

Memory layout:

```
matrix[0][0] matrix[0][1] matrix[0][2] ...
matrix[1][0] matrix[1][1] matrix[1][2] ...
```

#### Cache-friendly traversal

```c
for (int i = 0; i < 1000; i++) {
    for (int j = 0; j < 1000; j++) {
        sum += matrix[i][j];
    }
}
```

This follows the memory layout.

#### Cache-unfriendly traversal

```c
for (int j = 0; j < 1000; j++) {
    for (int i = 0; i < 1000; i++) {
        sum += matrix[i][j];
    }
}
```

This jumps through memory and causes cache misses.

## Memory Allocation Patterns

Frequent heap allocations can also hurt performance.

Example: 

```c
for (int i = 0; i < n; i++) {
    int *p = malloc(sizeof(int));
    *p = i;
}
```

This causes:

- allocator overhead
- memory fragmentation
- poor quality

**Better pattern:**

```c
int *array = malloc(n * sizeof(int));
```

Allocate once and reuse memory.

---

## Data Layout Matters

Let's consider two structure.

#### Array of structures

```c
typedef struct {
    float x;
    float y;
    float z;
} Point;

Point points[n]; // an array of size [n] of type Point
```

#### Structure of arrays

```c
float x[n]; // an array of size [n] of type float
float y[n];
float z[n];
```

For some workloads (especially SIMD operations), the second layout can be faster because data of the same type is contiguous.

This is an example of **data-oriented design**.

---

## Branch Prediction

Modern CPUs try to predict the outcome of branches.

Example:

```c
if (x > 0)
```

If the condition is predictable, execution is fast.

If the condition is random, performance may decrease due to **branch misprediction**.

Example: 

```c
if (random() % 2)
```

The CPU cannot predict this well.

## Predict Optimization

A famous quote by *Donald Knuth*:

> "Premature optimization is the root of all evil."

Meaning:

- first write **correct code**
- then **measure performance**
- optimize only where needed

Blind optimization often makes code:

- harder to read
- harder to maintain
- not actually faster

---

## Professional Performance Workflow

Typical performance workflow:

1. Write correct program
2. Measure performance
3. Identify bottleneck
4. Optimize critical sections

Tools used in real projects:

- **profilers (`perf`, `gprof`)**
- **benchmarking tools**
- **hardware counters**

Optimization without measurement is usually ineffective.

## Connection to the `csvstat` project

The `csvstat` project shows some performance-aware design:

Good decisions made:

- **streaming processing** (no full dataset in memory)
- **single line buffer reuse**
- **no extra allocations per row**
- **in-place parsing**

These patterns are typical in **high-performance data processing systems**.

---

## Takeaway

> - Performance in modern systems is often limited by **memory access**, not CPU speed.  
> - **Sequential memory access** is *much faster than random access*.  
> - CPU crashes reward programs with **good locality**.  
> - Avoid *unnecessary allocations* in *performance-critical code*.  
> - **Data layout** strongly affects performance.  
> - Always **measure before optimization**.  
> - Correctness comes before optimization.
