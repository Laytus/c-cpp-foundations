# Module 5.3 – Assertions and Contracts

### Learning Goals

- Understand what **assertions** are and why they are useful
- Learn the concept of **programming contracts**
- Distinguish between **assertions** and **error handling**
- Use assertions to enforce **internal invariants**
- Understand when assertions should and should not be used
- Learn how assertions behave in **debug vs release builds**

---

## The Core Idea

An **assertion** is a statement that verifies a condition that **must always be true** during program execution.

Example:

```c
assert(ptr != NULL);
```

This means:

> If this condition is `false`, the program is broken.

Assertions are used to verify **programmer assumptions**, not to handle normal runtime errors.

They are primarily used during **development and debugging** to *detect bugs early*.

---

## Assertions vs Error Handling

This distinction is extremely important.

| Situation | Use |
|-----------|-----|
| User input is invalid | Error Handling |
| File cannot be opened | Error Handling |
| Memory allocation fails | Error Handling |
| Internal invariant is violated | Assertion |

Example:

**Error Handling**

```c
if (!p) {
    return ERROR;
}
```

This is a **recoverable situation**.

**Assertion**

```c
assert(parser != NULL);
```

If `parser` is `NULL`, it means:

> the program logic is wrong

This is a **programming bug**, not a runtime condition.

---

## What is a Contract?

A **contract** defines the expected behavior between a function and its caller.

It usually contains three elements:

| Contract Element | Meaning |
|------------------|---------|
| Preconditions | What must be true before calling the function |
| Postconditions | What must be true after the function returns |
| Invariants | Conditions that must always remain true |

#### Preconditions

A **precondition** is a requirement the caller must satisfy.

Example:

```c
double mean(const Stats *s);
```

**Precondition:**

> `s` **must not** be `NULL`   
> `s` **must** contain at least one value

We can enforce this with assertions:

```c
assert(s != NULL);
assert(s->count > 0);
```

#### Postconditions

A **postconditions** verifies the result of a function.

Example:

```c
vector_push(v, x);
```

After the operation:

```
vector size increased by 1
```

Assertion example:

```c
size_t old_size = v->size;
vector_push(v, x);

assert(v->size == old_size + 1);
```

#### Invariants

An **invariant** is a condition that must always hold for a data structure.

Example: *vector invariant*

```
size <= capacity
```

Assertion:

```c
assert(v->size <= v->capacity);
```

Invariants are extremely important in complex systems.

---

## Assertion Example

Typical assertion usage:

```c
int vec_push(Vector *v, int x) {
    assert(v != NULL);
    assert(v->size <= v->capacity);

    v->data[v->size++] = x;

    assert(v->size <= v->capacity);
    return 0;
}
```

Assertions verify that the structure remains valid.

---

## Debug vs Release Builds

In C, assertions are usually disabled in **release builds**.

This is controlled by the *macro*:

```c
NDEBUG
```

If `NDEBUG` is defined before including `<assert.h>`, assertions are disabled.

Example:

```c
#define NDEBUG
#include <assert.h>
```

Then:

```c
assert(x > 0);
```

becomes:

```c
((void)0)
```

Meaning the assertion code dissapears.

---

## Custom Assertion Macros

Professional projects often implement their own assertion macros.

Example pattern:

```c
#define CSV_ASSERT(cond) \
    do { \
        if (!(cond)) { \
            fprintf(stderr, "Assertion failed: %s\n", #cond); \
            abort(); \
        } \
    } while (0)
```

Advantages:

- custom error messages
- better debugging output
- integration with project logging

This is exactly what is implemented in the `csvstat` project with:

```
CSVSTAT_ASSERT(...)
```

## Where Assertions should be used

Assertions are useful for checking:

- pointer validity
- internal invariants
- impossible states
- assumptions about program logic

Examples:

```c
assert(ptr != NULL);
assert(index < size);
assert(state != INVALID_STATE);
```

---

## Where Assertions shoud NOT be used

Do not use assertions for:

- user input validation
- file errors
- network errors
- memory allocation failure

These are **normal runtime failures**, not programming errors.

Example:

Bad: 

```c
assert(fp != NULL);
```

Correct:

```c
if (!fp) {
    return ERROR;
}
```

---

## Real Industry Usage

Assertions are heavily used in:

- operating systems
- database engines
- compilers
- game engines

Examples:

- Linux kernel (BUG_ON)
- SQLite internal assertions
- LLVM assertion framework

Assertions are essential for **maintaining large systems**.

---

## Practical Rule

Think of assertions as:

> "this must never happen"

If the condition could realistically fail during normal execution,
it should be **error handling instead**.

---

## Takeaway

> - *Assertions** verify internal assumptions in the program.  
> - They detect programming bugs early during *development*.  
> - Assertions enforce **contracts** (*preconditions, postconditions, invariants*).  
> - They should not replace normal error handling.  
> - Assertions are typically enabled in debug builds and removed in release builds.  
> - Large systems rely heavily on assertions to maintain correctness.
