# Module 7.3 – RAII (Resource Acquisition Is Initialization)

### Learning Goals

- Understand the concept of **RAII**
- Learn how C++ ties resource lifetime to object lifetime
- Understand **constructors** and **destructors**
- Learn how RAII prevents common bugs (leaks, forgotten cleanup)
- Recognize RAII patterns in modern C++ code

---

## The Core Idea

RAII means:

> Resource Acquisition Is Initialization

Which sounds strange at first, but it simply means:

> **Resources are acquired when an object is created and released when the object is destroyed.**

Examples of resources:

- memory
- files
- sockets
- mutexes
- database connections
- GPU buffers

Instead of managing these manually, C++ ties them to **object lifetime**.

---

## Why RAII exists

In C, resource management is manual.

Example: 

```c
FILE* f = fopen("data.txt", "r");
if (!f) return -1;

/* use file */

fclose(f);
```

Problems:

- easy to forget `fclose`
- cleanup scattered across code
- error paths leak resources

Example bug:

```c
FILE* f = fopen("data.txt", "r");
if (!f) return -1;

if (something_failed()) {
    return -2; // forgot fclose
}
```

This causes a **resource leak**.

---

## RAII Solution

In C++, we encapsulate the resource inside an object.

Example idea:

```cpp
File file("data.txt");
```

When the object is created:

```txt
constructor → opens file
```

When the object goes of scope:

```txt
destructor → close file
```

The programmer **cannot forget cleanup**.

---

## Constructors and Destructors

RAII relies on two special functions.

#### Constructor

Runs when an object is created.

```cpp
class File {
public:
    File(const std::string& name) {
        // acquire resource
    }
};
```

#### Destructor

Runs when the object is destroyed.

```cpp
~File() {
    // release resource
}
```

Destructors run automatically when:

- leaving a scope
- object lifetime ends
- stack unwinds after exceptions

---

## Example RAII Class

Example wrapper around a file.

```cpp
class File {
    FILE* handle;

public:

    File(const char* name) {
        handle = fopen(name, "r");
    }

    ~File() {
        if (handle) {
            fclose(handle);
        }
    }
};
```

Usage: 

```cpp
void process() {
    File file("data.txt");

    // use file

} // destructor automatically runs here
```

Even if an error occurs, the destructor still runs.

---

## Stack Unwinding

One of the biggest advantages of RAII appears during errors.

Example:

```cpp
void f() {
    Resource r;

    if (error) {
        return;
    }
}
```

Even when the function exits early:

> `r`'s destructor runs

This guarantees cleanup.

---

## RAII with the Standard Library

Modern C++ relies heavily on RAII.

Examples:

#### std::vector

```cpp
std::vector<int> v(100);
```

Internally:

> constructor → allocates memory
> destructor → frees memory

No `free()` needed.

#### `std::string`

```cpp
std::string s = "hello";
```

Memory management is automatic.

#### `std::unique_ptr`

```cpp
std::unique_ptr<int> p(new int(5));
```

When `p` is detroyed:

> delete automatically called

---

## RAII and Scope

RAII is strongly connected to scope.

Example:

```cpp
{
    std::vector<int> v(100);
}
```

When the block ends:

> `v` destroyed runs
> memory freed

This is deterministic.

Unlike garbage collection systems.

---

## RAII vs Garbage Collection

Languages like Java or Python rely on **garbage collection**.

Meaning:

> cleanup happens later
> time is unpredictable

C++ RAII:

> cleanup happens immediately
> when object leaves scope

This is extremely important for:

- real-time systems
- game engines
- trading systems
- HPC

---

## RAII prevents entire classes of bugs

RAII eliminates many common C bugs.

#### Memory leaks

```cpp
std::vector
std::string
unique_ptr
```

#### File leaks

```cpp
std::ifstream
std::ofstream
```

#### Lock leaks

```cpp
std::lock_guard<std::mutex>
```

Example:

```cpp
std::lock_guard<std::mutex>
```

Example:

```cpp
std::lock_guard<std::mutex> lock(m);
```

The `mutex` automatically unlocks when the scope ends.

Examples:


This is why modern C++ code **rarely uses manual memory management**.

## Mental Model

Think of RAII object as:

> resource guardians

They guarantee:

> resource acquired → constructor
> resource released → destructor

---

## Takeaway

> - **RAII** ties **resource lifetime to object lifetime**.  
> - *Constructor* acquire resources.  
> - *Destructors* release resources.  
> - Cleanup happens automatically when objects leave scope.  
> - RAII prevents leaks and forgotten cleanup.  
> - Most modern C++ abstraction rely on RAII.
