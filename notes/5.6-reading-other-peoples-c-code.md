# Module 5.6 – Reading Other People's C Code

### Learning Goals

- Develop strategies for understanding unfamiliar C codebases
- Learn how to identify the **entry point** of a program
- Understand how to reconstruct program architecture from source files
- Learn to recongnize common C patterns used in real systems
- Learn how to read C code **systematically instead of line-by-line**
- Build the skill of navigation large codebases efficiently

---

## The Core Idea

Professional programmers spend a large portion of their time **reading code written by other people**.

In many real jobs:

> 80–90% reading code
> 10–20% writing code

This includes:

- maintaining legacy systems
- fixing bugs
- extending existing features
- integrating libraries

Being able to quickly understad unfamiliar code is a **critical engineering skill**.

---

## The wrong way to read code

Beginners often read code **line by line from top to bottom**.

Example: 

```c
int main() {
    ...
}
```

They try to understand **every line immediately**.

This approach fails because:

- large systems contain thousands of lines
- details only make sense in context
- many functions are not relevant initially

Instead, we should read code **top-down and structurally**.

---

## The Correct Reading Strategy

Typical process:

> 1. *Identify* the **program entry point**   
> 2. *Understand* the **high-level architecture**   
> 3. *Identify* **main modules**   
> 4. *Explore* **key data structures**   
> 5. *Trace* important **execution paths**   
> 6. *Dive* into **implementation details**   

This mirrors how experienced engineers approach large systems.

---

## Step 1 – Find the Entry Point

Most C programs start with:

```c
int main(int argc, char **argv)
```

Understanding `main()` reveals:

- program inputs
- program workflow
- major components used

Example:

```c
int main(...) {
    parse_cli(...);
    open_file(...);
    process_data(...);
    print_results(...);
}
```

This quickly reveals the **high-level pipeline**.

---

## Step 2 – Identify the Modules

C projects are often organized into modules:

Example:

```
src/
    csv.c
    stats.c
    line_reader.c
```

Each module typically handles one responsability.

Example responsabilities (from the `csvstat` project):

```
line_reader → reading file lines
csv_parser  → aplitting CSV rows
stats       → statistical computation
```

This reveals the **architecture of the program**.

---

## Step 3 – Find the Key Data Structures

Most programs revolve around **core data structures**.

Example:

```c
typedef struct {
    double mean;
    double variance;
    size_t count;
} Stats;
```

Understanding the structures tells us:

- what data the system stores
- how the system evolves state
- what the program ultimately computes

---

## Step 4 – Trace the Execution Flow

After identifying the architecture, we follow the program flow.

Explain pipeline:

```
read line
→ parse CSV
→ extract column
→ parse number
→ update statistics
```

Tracing the path shows **how the program transforms input into output**.

This is far more useful than reading isolated functions.

---

## Step 5 – Recognize Common Patterns

Real C code frequently uses recurring patterns.

Examples:

**Resource cleanup**

```c
goto cleanup;
```

**Error propagation**

```c
if (result != 0) {
    return result;
}
```

**Defensive checks**

```c
if (!ptr) {
    return -1
}
```

**Module encapsulation**

```c
static functions
```

Recognize these patterns speeds up understanding dramatically.

---

## Step 6 – Ignore Details Initially

When first reading code:

Do **not** focus on:

```
pointer arithmetic
small helper functions
formatting details
```

Instead focus on:

```
architecture
data flow
major algorithms
```

Once the big picture is clear, the details become easier to understand.

---

## Professional Reading Techniques

Experienced developers use additional strategies.

#### Search for symbols

Tools like:

```
grep
ripgrep
```

Example:

```bash
grep -R "stats_update"
```

Finds where a function is used.

#### Jump to definitions

Editors like:

- VSCode
- CLion
- Vim
- Emacs

Allow jumping directly to function definitions.

This makes navigation much faster.

#### Read header files first

Header files (`.h`) reveal the **public interface**.

Example:

```c
int stats_update(Stats *s, double value);
```

Headers show:

- what a module exposes
- what functions exist
- what structure are defined

This often explains the design before reading the implementation.

## Applying this to the `csvstat` project

The project already has a clear modular structure:

```
app/
    main.c
src/
    csv.c
    line_reader.c
    stats.c
```

A reader should understand the program by:

1. Starting with `main.c`
2. Identifying the pipeline
3. Inspecting modules
4. Understanding the `Stats` structure

This is exactly how real engineers would approach the code.

---

## Takeaway

> - Reading code is a core engineering skill.  
> - Do not read code line-by-line initially.  
> - Start with **entry points and architecture**.  
> - Identify **modules and data structures**.  
> - Trace the **main execution flow**.  
> - Recognize *common C patterns*.  
> - Use tools like `grep` and *editor navigation*.  
> - Always understand the **big picture before the details**.
